# Phase 10: Cashier Billing Module

## Scope

Phase 10 migrates the cashier bill workflow from the generic module controller into a dedicated billing module while preserving the existing MVC direction:

- JavaFX view: `view/cashier/bill-editor.fxml`
- Controller: `BillController`
- Services: `BillingService`, `InventoryService`, `ReceiptService`
- Repositories: `BillRepository`, `BillItemRepository`, `ProductRepository`, `NotificationRepository`
- Model: `Bill`, `BillItem`, `BillItemRequest`

## Implemented Changes

- Added `BillController` for the cashier checkout scene.
- Replaced the bill editor FXML binding from `ModuleController` to `BillController`.
- Added editable product search/autocomplete through a product `ComboBox`.
- Added quantity and sector validation before a line item enters the bill.
- Added stock validation through `BillingService.prepareItem`.
- Added running total and current item count.
- Added remove selected item, clear bill, and finish bill actions.
- Added `BillItemRepository` to persist bill line items.
- Updated `BillingService.finishBill` so bill summary, bill items, stock decrease, and low-stock notifications happen in one database transaction.
- Updated `ProductRepository.decreaseQuantity` so the final stock update only succeeds when enough stock still exists.
- Kept receipt rendering/writing in `ReceiptService`, outside the database save operation.

## Transaction Flow

1. Validate cashier, receipt file name, and bill items.
2. Create the next bill id.
3. Save the bill summary.
4. Save all bill items.
5. Decrease product stock with an atomic `quantity >= requestedQuantity` guard.
6. Create low-stock notifications when needed.
7. Commit the database transaction.
8. Generate the receipt from the returned saved bill data.

If any database step fails, the transaction rolls back and the receipt is not generated by the controller.

## Database Notes

`BillItemRepository` creates a `bill_items` table when the active legacy database does not have one yet:

- `billid`
- `productname`
- `quantity`
- `unitprice`
- `linetotal`
- `sector`

It can also write/read the normalized v2 naming shape:

- `bill_id`
- `product_name`
- `quantity`
- `unit_price`
- `line_total`
- `sector`

## Acceptance Mapping

- Bill cannot complete if stock is insufficient: final stock decrement is guarded by SQL and throws `InsufficientStockException` when no row updates.
- Inventory and bill records update together: bill, bill items, stock, and notifications are inside one `DatabaseManager.inTransaction` call.
- Failed bill creation rolls back cleanly: runtime and SQL failures roll back through `DatabaseManager`.
- Receipt generation is separate from database save: `ReceiptService.writeReceipt` is called by `BillController` after `BillingService.finishBill` returns.
